01

Source Code — What Developers Write

Modern JavaScript/TypeScript projects are organized into hundreds or thousands of small files, each handling a specific concern:

project/
  src/
    entrypoints/cli.tsx       # Where the program starts
    main.tsx                   # Core application logic
    tools/
      BashTool/BashTool.tsx   # One feature = one module
      GrepTool/GrepTool.tsx
    services/
      api/claude.ts           # API communication
    components/
      StatusLine.tsx          # UI components
  node_modules/                  # Third-party dependencies (thousands of files)
  package.json                 # Project manifest
  tsconfig.json                # TypeScript configuration

TypeScript (.ts, .tsx) is a superset of JavaScript that adds static types. Browsers and Node.js can't run TypeScript directly — it must be transpiled to plain JavaScript first.

JSX/TSX is a syntax extension that lets you write HTML-like markup in JavaScript. It's used with React and gets transformed into function calls:

// What you write (TSX)
const StatusLine = () => <Box><Text color="green">Ready</Text></Box>;

// What the bundler outputs (JS)
const StatusLine = () => React.createElement(Box, null,
  React.createElement(Text, { color: "green" }, "Ready")
);
02

Bundling — Many Files Into One

A bundler takes your entire source tree (your code + all dependencies) and produces a single output file that can run directly. Think of it as compiling, but for JavaScript.

The bundling process
INPUT (4,756 files)                    OUTPUT (1 file)

src/entrypoints/cli.tsx  ──┐
src/main.tsx             ──┤
src/tools/BashTool.tsx   ──┤
src/services/api/*.ts    ──┤          ┌──────────────────┐
src/components/*.tsx     ──┼── bundler ──> │  cli.js (13 MB)   │
node_modules/lodash-es/* ──┤          │  16,667 lines     │
node_modules/ink/*       ──┤          └──────────────────┘
node_modules/react/*     ──┤
... 4,748 more files     ──┘

What the bundler does

1

Resolves imports — Starting from the entry point, follows every import and require() statement recursively to discover all needed files.

2

Transpiles — Converts TypeScript to JavaScript, transforms JSX to function calls, and downgrades modern syntax for older Node.js versions if needed.

3

Inlines — Copies the code from every dependency directly into the output file. The result is self-contained with zero runtime dependencies.

4

Tree-shakes — Removes unused exports (dead code elimination). If you only use 3 functions from lodash, only those 3 end up in the bundle.

Common bundlers

BundlerSpeedUsed By
BunVery fast (Zig-based)Claude Code
esbuildVery fast (Go-based)Vite (dev mode)
webpackSlower, very configurableMost older projects
RollupMedium, clean outputLibraries, Vite (prod)
TurbopackFast (Rust-based)Next.js
Why bundle?

Without bundling, you'd need to ship your entire node_modules folder (often 500MB+) and all your source files. A bundle is smaller, faster to install, and has zero dependency conflicts.

03

Source Maps — The Debugging Bridge

When you bundle 4,756 files into one, debugging becomes impossible. An error at line 14,322 of cli.js tells you nothing. Source maps solve this by mapping bundled positions back to original files.

How source maps work
cli.js line 14322, col 8
      │
      │  source map lookup
      ▼
src/tools/BashTool.tsx line 47, col 4

A source map is a JSON file with this structure:

{
  "version": 3,
  "sources": [                           // original file paths
    "../src/main.tsx",
    "../src/tools/BashTool.tsx",
    "../node_modules/react/index.js"
  ],
  "sourcesContent": [                    // full source of each file (!)
    "import React from 'react';\n...",
    "export function BashTool() {\n...",
    "'use strict';\nvar React = ..."
  ],
  "mappings": "AAAA,SAAS,QAAQ...",  // encoded position data
  "names": ["React", "useState", ...]   // variable names
}

Source map modes

bun build --sourcemap=none

No source map generated. Smallest output. Stack traces reference the bundled file only. Best for production if you don't need debugging.

bun build --sourcemap=external
# Produces: cli.js + cli.js.map
# cli.js ends with:
//# sourceMappingURL=cli.js.map

Separate .map file. The bundle references it with a trailing comment. This is what Claude Code used — and the .map file got published.

bun build --sourcemap=inline
# cli.js ends with:
//# sourceMappingURL=data:application/json;base64,eyJ2ZXJzaW9uIjoz...

The entire source map is base64-encoded and appended to the bundle. Makes the JS file much larger. Useful for development only.

bun build --sourcemap=external
# Then strip the reference comment from cli.js
# Keep cli.js.map on your error tracking server only

Generate the map but don't link it in the bundle. Upload it to an error tracking service (like Sentry) for server-side symbolication. The map never ships to users.

The danger of sourcesContent

The sourcesContent field embeds the complete original source code of every file. If you publish a source map with this field, anyone can reconstruct your entire codebase with a simple script. This is exactly what happened with Claude Code.

package.json — The Manifest

Every npm package has a package.json. It's the single source of truth for what the package is, what it does, and how it connects to the ecosystem.

{
  // Identity
  "name": "@anthropic-ai/claude-code",  // scoped: @org/package
  "version": "2.1.88",                 // semver: major.minor.patch
  "description": "CLI for Claude AI",
  "license": "SEE LICENSE IN README.md",

  // Entry points
  "bin": { "claude": "cli.js" },       // CLI executable
  "main": "index.js",                  // require('package')
  "module": "index.mjs",               // import from 'package'
  "type": "module",                    // ESM by default

  // Runtime requirements
  "engines": { "node": ">=18.0.0" },

  // Dependencies (see chapter 8)
  "dependencies": {},
  "optionalDependencies": {},
  "devDependencies": {},

  // Lifecycle scripts
  "scripts": {
    "build": "bun build src/entrypoints/cli.tsx --outfile cli.js",
    "test": "vitest",
    "prepare": "..."    // runs before publish
  },

  // What files to include in the package
  "files": ["cli.js", "vendor/"]        // <-- Claude Code was MISSING this
}

Key fields explained

bin

Maps command names to scripts. When installed globally, npm creates a symlink so you can type claude in your terminal:

/usr/local/bin/claude  →  /usr/local/lib/node_modules/@anthropic-ai/claude-code/cli.js

type

JavaScript has two module systems. This field picks the default:

ValueSystemSyntax
"module"ESM (modern)import x from 'y'
"commonjs"CommonJS (legacy)const x = require('y')

version (semver)

Versions follow Semantic Versioning: MAJOR.MINOR.PATCH

  • MAJOR (2.x.x) — Breaking changes. Your code may need updates.
  • MINOR (x.1.x) — New features, backwards compatible.
  • PATCH (x.x.88) — Bug fixes only.
05

npm pack — What Goes in the Box

npm pack creates the tarball (.tgz archive) that gets uploaded to the registry. This is the critical step where you control what ships to users.

File selection priority

npm uses the first mechanism it finds:

1

"files" field in package.json — An allowlist. Only listed files/dirs are included. This is the most secure approach. package.json, README, and LICENSE are always included regardless.

"files": ["cli.js", "sdk-tools.d.ts", "vendor/"]
2

.npmignore file — A denylist, like .gitignore but for npm. Files matching these patterns are excluded.

*.map
*.test.ts
src/
tsconfig.json
3

.gitignore fallback — If neither "files" nor .npmignore exists, npm uses .gitignore to decide what to exclude.

Allowlist (safe)

// package.json
"files": [
  "cli.js",
  "vendor/"
]

Only what you list gets in. New files are excluded by default. Explicit. Safe.

No filter (dangerous)

// package.json
// no "files" field

// no .npmignore

// → EVERYTHING ships

Every file in the directory goes into the tarball. Source maps, env files, tests — all of it.

Preview before publishing

# See what would be packed without creating the tarball
$ npm pack --dry-run

npm notice Tarball Contents
npm notice 13.0MB  cli.js
npm notice 59.8MB  cli.js.map      # ← you'd see this and catch it
npm notice 1.2kB   package.json
Best practice

Always use the "files" allowlist in package.json. Always run npm pack --dry-run before publishing. Add it as a CI check.

06

npm publish — Shipping to the World

Publishing uploads your tarball to the npm registry, making it available to anyone via npm install.

npm publish
→
run "prepare" script
→
npm pack (create .tgz)
→
upload to registry
→
available to install

Lifecycle scripts

npm runs scripts at specific moments during publish:

ScriptWhenCommon use
prepublishOnlyBefore pack, only on publishRun tests, lint
prepareBefore pack (and after install)Build step, safety checks
prepackBefore tarball is createdGenerate types
postpackAfter tarball is createdCleanup
postpublishAfter uploadNotifications

Claude Code uses prepare as a publishing guard:

"prepare": "node -e \"if (!process.env.AUTHORIZED) { console.error('ERROR: Direct publishing is not allowed.'); process.exit(1); }\""

This prevents a developer from accidentally running npm publish from their laptop. Only their CI/CD pipeline (which sets AUTHORIZED=1) can publish.

Scoped packages

The @anthropic-ai/ prefix is a scope. It namespaces packages to an organization:

@anthropic-ai/claude-code    # Anthropic owns this scope
@types/react                  # DefinitelyTyped types
@img/sharp-linux-x64           # Sharp image library
lodash                        # unscoped (anyone can publish)

npm install — The Consumer Side

When someone runs npm install -g @anthropic-ai/claude-code, here's what happens:

Installation flow
$ npm install -g @anthropic-ai/claude-code
  │
  ├─ 1. GET registry.npmjs.org/@anthropic-ai/claude-code
  │     → metadata: version, tarball URL, dependencies
  │
  ├─ 2. Download tarball (31.2 MB compressed)
  │
  ├─ 3. Extract to /usr/local/lib/node_modules/@anthropic-ai/claude-code/
  │     → 20 files, 102.8 MB
  │     → includes cli.js.map (the leaked source)
  │
  ├─ 4. Install optionalDependencies
  │     → @img/sharp-linux-x64 (matches this platform)
  │     → @img/sharp-darwin-arm64 (skipped: wrong platform)
  │
  └─ 5. Create symlink
        /usr/local/bin/claude → cli.js

Global vs local install

CommandWhereUse case
npm install -g pkg System-wide (/usr/local/) CLI tools you run from anywhere
npm install pkg Project-local (./node_modules/) Libraries used in your code

The node_modules directory

node_modules/ is where npm puts installed packages. It's a flat directory (since npm v3) where each package gets its own folder:

node_modules/
  @anthropic-ai/
    claude-code/
      cli.js
      cli.js.map        # anyone can read this
      package.json
      vendor/
        ripgrep/
        audio-capture/
  @img/
    sharp-linux-x64/    # optional dep (only on Linux x64)

Dependencies — The Dependency Graph

Dependencies are other packages your code needs. There are three types, and understanding them is critical:

TypeWhen installedPurpose
dependencies Always (by you AND your consumers) Code needed at runtime
devDependencies Only during development Testing, building, linting tools
optionalDependencies If possible (failures are OK) Platform-specific native code

Claude Code's strategy: zero dependencies

"dependencies": {},                    // nothing!
"optionalDependencies": {
  "@img/sharp-darwin-arm64": "^0.34.2", // image processing
  "@img/sharp-linux-x64": "^0.34.2",
  // ... 7 more platforms
}

Because everything is bundled into cli.js, there are no runtime dependencies. This means:

  • No dependency conflicts with other packages on your system
  • No supply-chain attacks via transitive dependencies
  • No version resolution issues
  • Installs in seconds (just one download)

The sharp libraries are optional because they're native binaries (compiled C/C++) that differ per OS and CPU architecture. npm only installs the one matching your system.

Version ranges

Dependencies use semver ranges to specify which versions are acceptable:

SyntaxMeaningExample range
^0.34.2Compatible (same major)0.34.2 ≤ v < 0.35.0
~0.34.2Patch updates only0.34.2 ≤ v < 0.35.0
0.34.2Exact versionOnly 0.34.2
*Any versionAnything
>=18.0.0Minimum version18.0.0 and above

Lock files

A lock file records the exact versions that were installed, so every developer (and CI) gets identical dependencies:

ToolLock file
npmpackage-lock.json
yarnyarn.lock
pnpmpnpm-lock.yaml
bunbun.lock
Spotted in the package

Claude Code ships a bun.lock in the published package — this reveals they use Bun as their package manager and build tool. It shouldn't be in the published package (another file that should've been excluded).

Case Study — The Claude Code Leak

Let's put it all together. Here's exactly how Anthropic's closed-source CLI had its full source code leaked via a 60MB source map in a public npm package.

The leak chain
DEVELOPMENT                        BUILD                         PUBLISH

 2,374 TypeScript files              Bun bundler                    npm pack
 in private repo                     with --sourcemap=external
                                                                    no "files" field
 src/main.tsx  ─────┐                                               no .npmignore
 src/tools/*   ─────┤           ┌─ cli.js      (13 MB)  ──┐
 src/services/*─────┼── build ──┤                          ├── pack ── publish
 node_modules/* ────┤           └─ cli.js.map  (60 MB)  ──┘
 (4,756 files) ─────┘               ▲                         ▲
                                     │                         │
                               contains full            no filter =
                               sourcesContent           everything ships


RESULT: Anyone can run this to extract the full source:

 $ npm pack @anthropic-ai/claude-code
 $ tar xzf *.tgz
 $ node -e "
     const sm = JSON.parse(fs.readFileSync('package/cli.js.map'));
     sm.sources.forEach((path, i) => {
       fs.mkdirSync(dirname(path), { recursive: true });
       fs.writeFileSync(path, sm.sourcesContent[i]);
     });
   "
 → 4,756 files, 64 MB of readable source code

What was exposed

CategoryCountExamples
Application source1,332 .ts + 552 .tsxAll tools, services, components, commands
Internal packages~50 from @ant/*Chrome MCP, computer-use MCP
Dependencies source2,850 filesAll bundled node_modules
ArchitectureFullState management, API layer, auth flow
Unreleased featuresDozensCommands gated behind feature flags

Root cause

Three missing safeguards, any one of which would have prevented the leak:

1

No "files" allowlist in package.json — so npm included everything in the build directory.

2

No .npmignore — no denylist to catch the .map file.

3

No CI check to verify the tarball contents before publishing (e.g., npm pack --dry-run and assert no .map files).

Prevention — How to Not Leak Your Source

The checklist

Before every publish
  1. Use "files" allowlist in package.json — not .npmignore
  2. Build with --sourcemap=none for production, or use --sourcemap=hidden and upload maps to your error tracker only
  3. Run npm pack --dry-run in CI and fail if unexpected files appear
  4. Check unpacked size — if it's way bigger than expected, investigate
  5. Never include: .map, .env, lock files, test files, source directories

Secure package.json template

{
  "name": "@your-org/your-package",
  "version": "1.0.0",
  "files": [
    "dist/",          // only the build output
    "!dist/**/*.map"  // explicitly exclude source maps
  ],
  "scripts": {
    "build": "bun build src/index.ts --outfile dist/index.js --sourcemap=none",
    "prepublishOnly": "npm run build && npm pack --dry-run"
  }
}

CI check script

#!/bin/bash
# Add to your CI pipeline before npm publish

echo "Checking package contents..."
npm pack --dry-run 2>&1 | tee /tmp/pack-contents.txt

# Fail if any source maps are included
if grep -q '\.map' /tmp/pack-contents.txt; then
  echo "ERROR: Source maps detected in package!"
  exit 1
fi

# Fail if .env or credentials files are included
if grep -qE '\.env|credentials|\.key' /tmp/pack-contents.txt; then
  echo "ERROR: Sensitive files detected in package!"
  exit 1
fi

echo "Package contents look clean."
